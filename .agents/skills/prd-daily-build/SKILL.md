# PRD 기반 일일 빌드 스킬

## 언제 사용하나
- 사용자가 "Day N 시작", "오늘 진행", "다음 날짜 작업" 등을 요청할 때
- docs/PRD.md 부록 A(Day1~10 일정표)를 기준으로 매일 정해진 범위만 구현할 때

## 핵심 원칙 (반드시 준수)
1. **한 번에 전체를 구현하지 않는다.** 반드시 지정된 Day의 범위만 구현한다. 사용자가 명시적으로 요청하지 않는 한 여러 Day를 미리 땅안아 구현하지 말 것.
2. **이전 Day의 DoD(완료조건)가 충족되었는지 먼저 확인한다.** PROGRESS.md에서 이전 Day가 체크되어 있지 않다면, 진행 전에 사용자에게 확인한다.
3. **범위를 벗어나는 요청은 확인 후 진행한다.** 예: Day3 작업 중 Day7 기능(공유 기능)을 요청받으면, 범위 밖임을 알리고 진행 여부를 물어본다.
4. **각 Day 작업이 끝나면 반드시 git 커벋(필요 시 push)한다.** 커벋 메시지는 "Day N: <DoD 요약>" 형식을 따른다.
5. **PROGRESS.md를 갱신한다.** 해당 Day 항목을 완료 표시하고, 다음 Day 작업을 사전에 시작하지 않는다.
6. **마일스톤(M1~M5)이 걸린 Day는 해당 목표가 달성되었는지 명시적으로 확인 후 보고한다.**
7. **화면 작업 시 항상 docs/DESIGN_GUIDE.md를 먼저 확인하고 그 스타일(톤앤매너)을 엄격히 준수한다.**

## Git 워크플로
- 각 Day 작업 시작 전: `git status`로 현재 상태 확인
- 작업 완료 후: `git add -A && git commit -m "Day{N}: {DoD 요약}"`
- 원격 저장소가 연결되어 있다면: `git push origin <branch>`
- 브랜치 전략은 자유이나, 단순 구조로 눈 뒤에는 main/master에 직접 커벋해도 무방하다.
